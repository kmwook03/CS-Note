### 1. 저장 매체 (Storage Media)

대용량 데이터를 효율적으로 저장하고 관리하기 위한 물리적 계층 구조와 디스크의 특성을 다룹니다.

*   **메모리 계층 구조 (Memory Hierarchy):** 캐시(Cache) > 메인 메모리(Main Memory) > 보조 기억 장치(SSD, HDD) > 3차 기억 장치(Tape) 순으로 구성되며, 위로 갈수록 속도가 빠르고 비싸며, 아래로 갈수록 용량이 큽니다.
*   **SSD의 특성:** 비대칭적인 읽기/쓰기 속도를 가지며, 덮어쓰기(Overwrite)가 불가능해 블록 단위로 삭제(Erase) 후 페이지 단위로 써야 하는 한계와 수명(Aging) 문제가 있습니다. 
*   **하드 디스크(HDD) 접근 시간 (Disk Access Time):** 디스크 데이터 접근 시간 $t$는 $t_S$ (탐색 시간: 헤드가 올바른 트랙으로 이동하는 시간) + $t_R$ (회전 지연: 올바른 섹터가 헤드 아래로 올 때까지의 시간) + $t_T$ (전송 시간)의 합으로 계산됩니다.
*   **블록 단위 접근 (Block-Oriented Access):** 디스크 I/O는 바이트 단위가 아닌 '블록(Block)' 단위로 이루어지며, 디스크 접근 횟수를 줄이기 위해 메인 메모리에 버퍼를 두어 적중률(Hit Ratio)을 높이는 것이 성능의 핵심입니다.

### 2. 파일 구조와 OS 지원 (File Organization & OS Support)

운영체제가 파일 시스템을 어떻게 블록과 매핑하는지, 파일 내 레코드들을 어떻게 구성하는지 배웁니다.

*   **i-node (인덱스 노드):** 파일의 속성(크기, 소유권 등)과 실제 데이터 블록을 가리키는 포인터들을 저장하는 자료구조입니다. `open()`, `read()` 함수 호출 시 OS는 루트부터 경로를 따라 i-node를 찾고, 접근 권한을 확인한 뒤 OpenFileTable에 등록하여 파일 디스크립터(fd)를 반환합니다.
*   **레코드 구성 방식:**
    *   **고정 길이 레코드 (Fixed Size Record):** 구현이 쉽고 배열처럼 `(n-1) * 레코드 크기`로 디스크 주소를 계산할 수 있습니다.
    *   **가변 길이 레코드 (Variable Length Record):** 필드 개수나 크기가 가변적일 때 사용합니다. 구분자(Delimiter)를 쓰거나, 한 페이지 내에서 단편화를 막고 빈 공간을 관리하는 **슬롯 페이지(Slotted Page)** 구조를 사용합니다. 블록 크기보다 큰 이미지/비디오 데이터는 BLOB(Binary Large Object)으로 처리합니다.

### 3. I/O 및 버퍼 관리 (I/O and Buffer Management)
디스크의 느린 속도를 보완하기 위해 메인 메모리 영역인 버퍼를 어떻게 관리하는지 다룹니다.

*   **버퍼 매니저:** CPU가 데이터를 요청하면 먼저 버퍼(메인 메모리)에 있는지 확인하고, 없으면 디스크에 접근하여 버퍼로 로드한 뒤 데이터를 제공합니다. 버퍼가 꽉 찼다면 페이지 교체(Replacement)가 일어납니다.
*   **교체 정책 (Replacement Policy):**
    *   **LRU (Least Recently Used):** 가장 오랫동안 사용되지 않은 블록을 교체하는 가장 일반적인 방식입니다.
    *   **MRU (Most Recently Used):** 최근에 사용된 것을 바로 교체합니다. 특정 접근 패턴(예: 반복적인 조인 연산 등)에서 유리할 수 있습니다.
    *   **Pinned block:** 디스크로 쫓겨나면 안 되는 중요한 블록을 메모리에 고정시키는 기법입니다.

### 4. 인덱싱(Indexing)과 Blocking Factor

#### 💡 Blocking Factor ($B_f$)
*   **개념:** **하나의 블록(Block) 안에 온전히 들어갈 수 있는 레코드의 개수**를 의미합니다. 파일 처리에서 가장 비용이 큰 작업은 '디스크 I/O(블록 접근)'이므로, 한 번 디스크를 읽어올 때 최대한 많은 레코드를 가져오는 것이 이득입니다.
*   **수식 및 계산 원리:** 
    *   **총 디스크 접근 횟수 ($N_D$) = 총 레코드 수 ($N_{record}$) / 블로킹 팩터 ($B_f$)**.
    *   예를 들어, 하나의 블록 크기가 4,000바이트이고, 레코드 1개의 크기가 400바이트라고 가정해 봅시다. 
    *   이때 $B_f$ = 4000 / 400 = 10이 됩니다 (한 블록에 10개의 레코드가 들어감).
    *   만약 총 1,000개의 레코드가 있다면, 이를 모두 순차 검색하기 위해 필요한 총 디스크 접근 횟수($N_D$)는 1000 / 10 = **100번** 입니다.
*   **결론:** 블로킹 팩터를 최대화(Maximize)하면 디스크 접근 횟수($N_D$)를 획기적으로 줄일 수 있어 전체적인 1단계 검색 속도(Phase 1)를 향상시킬 수 있습니다.

#### 트리 기반 인덱스
*   **B-tree:** 트리 높이의 균형이 맞춰진 트리로, 최악의 경우에도 $O(\log n)$의 탐색 속도를 보장합니다. 꽉 찬 노드에 새 데이터가 들어오면 **상향 분할(Upward Split)** 을 수행하여 균형을 유지합니다. 하지만 범위 검색(Range Search)에는 취약하다는 단점이 있습니다.
*   **B+-tree:** B-tree의 단점을 극복하기 위해, **모든 실제 데이터는 리프 노드(단말 노드)에만 저장** 하고, 리프 노드끼리 **연결 리스트(Linked List)** 로 묶은 형태입니다. 정확한 일치 검색뿐만 아니라 부등호(`>`, `<`)가 들어가는 **범위 검색(Range Search)** 에서 압도적으로 뛰어난 성능을 발휘합니다.

### 5. 해싱 (Hashing)
키 값을 통해 물리적 디스크 블록 번호를 직접 계산($h(v)=b$)하는 방식으로, 별도의 인덱스 데이터 구조 탐색(O(log n)) 없이 빠른 검색이 가능합니다. (범위 검색은 불가, Exact match 전용).

*   **정적 해시 (Static Hash):** 사용 가능한 블록 개수가 고정되어 있습니다. 한 버킷에 데이터가 넘치는 **오버플로우(Overflow)** 가 발생하면 오버플로우 체인(연결 리스트)을 만들거나 다음 빈 블록에 할당합니다.
*   **동적 해시 (Dynamic Hash / Extendable Hash):** 데이터베이스 크기 증감에 대응할 수 있습니다. 해시 값의 특정 길이(Prefix, $i$ bits)만 사용하여 버킷 주소 테이블($2^i$ 크기)을 유지합니다. 오버플로우가 발생하면 $i$값을 증가시켜 주소 테이블을 늘리고(Split) 데이터를 재배치합니다.

### 6. 다차원 인덱싱 (Multi-dimensional Indexing)
"키 > 180cm AND 몸무게 < 70kg"처럼 여러 속성(다차원)을 동시에 검색하는 기법입니다.

*   **확장 해싱 기법 - Grid File:** 차원을 격자(Grid)로 나누어 블록 포인터와 매핑하지만, 데이터가 없는 텅 빈 영역인 **Dead Space**가 발생하여 디스크 공간이 낭비되는 단점이 있습니다.
*   **확장 B+-tree 기법 - R-tree:** 데이터를 포함하는 **최소 경계 사각형(MBR, Minimum Bounding Rectangle)** 을 노드로 구성하는 트리입니다. 분할 시 면적(Area)이나 둘레(Perimeter), 겹치는 영역 크기를 최소화하는 방식으로 나누며 겹침(Overlapping)이 허용됩니다.
*   **R\*-tree (R-Star Tree):** R-tree의 개선판으로, 오버플로우 발생 시 즉시 분할하지 않고 데이터를 노드에서 삭제 후 **강제 재삽입(Re-Insertion Strategy)** 을 수행하여 MBR을 더 촘촘(Compact)하게 만드는 방식입니다.
