## CPU Virtualization

### 1. 프로세스 (Process)와 제한적 직접 실행 (Limited Direct Execution, LDE)

**프로세스의 개념 및 상태**
*   **정의:** 프로세스는 실행 중인 프로그램(A running program)을 의미합니다.
*   **머신 상태 (Machine State):** 프로세스는 메모리(주소 공간: Code, Data, Heap, Stack)와 CPU 문맥(PC, 스택 포인터 등 레지스터 값)으로 구성됩니다.
*   **특성:** 최신 시분할(Time-shared) 시스템에서 보호와 작업의 기본 단위가 되며, PID(Process ID)를 통해 식별됩니다. 주로 I/O 작업에 시간을 쏟는 **I/O-bound process**와 연산에 시간을 쏟는 **CPU-bound process**로 나뉩니다.

**제한적 직접 실행 (Limited Direct Execution)**
*   **CPU 가상화:** 수백 개의 프로세스가 동시에 실행되는 것처럼 환상을 제공하기 위해 시분할(Time sharing) 기법을 사용합니다.
*   **동작 방식:** 운영체제는 부팅 시 트랩 테이블(Trap table)을 초기화하여 시스템 콜이나 타이머 인터럽트 핸들러의 주소를 기억해 둡니다. 타이머 인터럽트를 통해 OS가 주기적으로 제어권을 되찾으며(Preemptive), 여러 프로세스를 번갈아 가며 실행(Interleaving)할 수 있게 합니다.


### 2. 문맥 교환 (Context Switch)

문맥 교환은 현재 실행 중인 프로세스를 멈추고 다른 프로세스로 CPU 제어권을 넘기는 저수준의 어셈블리 코드 동작입니다.
*   **저장(Save):** 현재 프로세스의 범용 레지스터 값, PC(Program Counter), 커널 스택 포인터 등을 해당 프로세스의 **커널 스택(Kernel stack)**에 저장합니다.
*   **복원(Restore):** 곧 실행될 프로세스의 커널 스택에서 저장해두었던 레지스터 값들을 복원합니다.
*   **스위치(Switch):** 곧 실행될 프로세스의 커널 스택으로 스위칭하여 실행을 재개합니다.


### 3. CPU 스케줄링 평가 지표 (Scheduling Metrics)

스케줄러의 품질을 측정하기 위해 다음과 같은 평가 지표가 사용되며, 주로 응답 시간의 최소화 및 처리량 극대화 등을 목표로 합니다.

1.  **반환 시간 (Turnaround Time):** 작업이 시스템에 도착한 시간부터 완료될 때까지 걸린 총시간입니다.
    *   **계산식:** $T_{turnaround} = T_{completion} - T_{arrival}$
2.  **응답 시간 (Response Time):** 작업이 도착한 후 처음으로 스케줄링(실행)되기까지 걸린 시간으로, 대화형(Interactive) 시스템에서 특히 중요합니다.
    *   **계산식:** $T_{response} = T_{firstrun} - T_{arrival}$
3.  **대기 시간 (Waiting Time):** 프로세스가 준비 큐(Ready Queue)에서 실행을 기다리며 보낸 총시간입니다.
4.  **기타 지표:**
    *   **CPU 활용률 (CPU Utilization):** CPU를 최대한 바쁘게 유지하는 비율.
    *   **처리량 (Throughput):** 단위 시간당 실행을 완료한 프로세스의 수.
    *   **공정성 (Fairness):** 모든 프로세스가 공평하게 CPU를 할당받는 정도 (성능 지표와 상충하는 경우가 많음).


### 4. 스케줄링 알고리즘의 특징 및 장단점

스케줄링은 프로세스가 자발적으로 CPU를 양보할 때까지 기다리는 **비선점형(Non-preemptive)** 과 OS가 타이머 등을 이용해 강제로 CPU를 빼앗는 **선점형(Preemptive)** 으로 나뉩니다.

#### ① FCFS (First-Come, First-Served)
*   **특징:** 가장 먼저 CPU를 요청한 프로세스에 먼저 할당하는 방식(FIFO)입니다.
*   **장점:** 매우 단순하고 구현이 쉽습니다. 모든 작업이 평등하게 취급되어 기아(Starvation) 현상이 없습니다.
*   **단점 (비선점형):** 평균 반환 시간이 최적화되지 않으며, 실행 시간이 긴 프로세스 뒤에 짧은 프로세스들이 기다려야 하는 **호위 효과(Convoy Effect)** 가 발생하여 평균 대기 시간이 길어질 수 있습니다.

#### ② SJF (Shortest-Job-First)
*   **특징:** 다음 CPU 버스트(실행 시간)가 가장 짧은 프로세스에 먼저 CPU를 할당합니다.
*   **장점:** 모든 작업이 동시에 도착한다고 가정할 때 평균 반환 시간과 대기 시간을 최소화하는 **최적(Optimal)** 의 알고리즘입니다.
*   **단점 (비선점형):** 프로세스의 다음 CPU 실행 시간을 미리 알기가 매우 어렵습니다 (과거 이력을 바탕으로 지수 평균 등을 사용해 예측해야 함).

#### ③ STCF (Shortest Time-to-Completion First / PSJF)
*   **특징:** SJF의 **선점형(Preemptive)** 버전입니다. 작업이 각기 다른 시간에 도착할 수 있는 환경에서, 새로운 작업이 도착하면 현재 남은 실행 시간과 비교하여 가장 짧게 남은 작업을 스케줄링합니다.
*   **장단점:** 반환 시간 측면에서는 뛰어나지만, 응답 시간 측면에서는 여전히 좋지 않은 성능을 보입니다.

#### ④ RR (Round-Robin)
*   **특징:** 각 프로세스에 타임 슬라이스(Time quantum, 보통 10~100ms)를 부여하고, 이 시간이 지나면 선점하여 준비 큐의 맨 뒤로 보내는 타임 슬라이싱(Time Slicing) 기법입니다.
*   **장점 (선점형):** 기아 현상이 없고 공정하며, **응답 시간이 매우 우수**합니다.
*   **단점:** SJF 등에 비해 평균 반환 시간은 나쁩니다. 타임 퀀텀의 크기가 성능을 크게 좌우하며, 너무 짧으면 문맥 교환 비용(Context switch overhead)이 커지고, 너무 길면 FCFS처럼 작동하여 응답성이 떨어집니다. (경험 법칙: CPU 버스트의 80%는 타임 퀀텀보다 짧아야 함).

#### ⑤ Priority Scheduling (우선순위 스케줄링)
*   **특징:** 각 프로세스에 우선순위(보통 가장 작은 정수가 가장 높은 우선순위)를 부여하여 높은 우선순위를 가진 프로세스에 CPU를 먼저 할당합니다. SJF도 우선순위가 '남은 실행 시간'인 우선순위 스케줄링의 일종입니다.
*   **단점:** 우선순위가 낮은 프로세스는 영원히 CPU를 할당받지 못하는 **기아 현상(Starvation)** 이 발생할 수 있습니다 (해결책으로 에이징(Aging) 사용).

#### ⑥ MLFQ (Multilevel Feedback Queue)
*   **특징:** 프로세스의 과거 실행 이력을 바탕으로 미래를 예측하여, 사전 정보 없이도 반환 시간을 최적화하고 대화형 작업의 응답 시간을 최소화하는 스케줄러입니다.
*   **해결 기법 (장점):**
    *   **우선순위 상향(Priority Boost):** 일정 시간(S)이 지나면 모든 작업을 최상위 큐로 이동시켜 **기아 현상을 방지**하고 I/O 바운드 프로세스로 변한 작업에 대응합니다.
    *   **더 나은 회계(Better Accounting):** 타임 슬라이스를 소진하기 직전에 I/O를 발생시켜 스케줄러를 속이는 행위(Gaming the scheduler)를 막기 위해, 해당 큐에서 할당된 시간을 모두 소진하면 우선순위를 강등시킵니다.

#### ⑦ Proportional Share Scheduling (비례 배분 스케줄링)
*   **특징:** 전체 CPU 시간을 T라고 할 때, 각 프로세스에 N개의 지분(Shares, N < T)을 할당하여 N/T 비율만큼의 CPU 시간을 보장하는 스케줄링입니다.
*   **특징:** 새로운 프로세스가 자원을 초과하여 요청할 경우 수용을 거부하는 허가 제어(Admission-control)를 수행합니다.

#### ⑧ Real-time Scheduling (실시간 스케줄링)
*   주기적(Periodic)으로 실행되며 일정한 마감일(Deadline)이 있는 프로세스들을 위한 알고리즘입니다.
*   **Rate-Monotonic:** 주기가 짧을수록 높은 우선순위를 부여하는 정적 우선순위 방식입니다.
*   **EDF (Earliest Deadline First):** 마감일이 가장 가까운 프로세스에 가장 높은 우선순위를 부여하는 동적 우선순위 방식입니다.

## Memory Virtualization

### 1. Memory Virtualization (메모리 가상화) 및 Address Space (주소 공간)
*   **개념:** 운영체제(OS)가 물리적 메모리를 가상화하여, 각 프로세스마다 크고 연속적인 자신만의 독자적인 메모리 공간(Address Space)을 가지고 있다는 환상을 제공하는 기법입니다.
*   **목표 (투명성):** 프로세스는 메모리가 다른 프로세스와 공유되고 있다는 사실을 인지하지 못하며, 프로그래밍에 편리한 추상화를 제공받습니다.

### 2. Base-bound (동적 재배치, Dynamic Relocation)
하드웨어(MMU)를 기반으로 모든 메모리 참조 명령어에 대해 동적으로 주소 변환을 수행하는 가장 기본적인 방식입니다.
*   **주소 변환 공식:** 
    *   **`물리 주소(Physical Address) = 가상 주소(Virtual Address) + Base 레지스터 값`**
*   **특징 및 보호(Protection):** 
    *   **Base 레지스터**는 해당 프로세스 물리 주소의 가장 작은(시작) 값을 가집니다.
    *   **Bound(Limit) 레지스터**는 논리 주소의 크기(범위)를 가집니다. 
    *   MMU는 가상 주소가 Bound 레지스터 값보다 작은지 확인하고, 크다면 하드웨어 예외(Exception/Trap)를 발생시켜 보호합니다.
*   **단점 (비효율성):** 프로세스의 스택(Stack)과 힙(Heap) 사이에 있는 **사용하지 않는 빈 공간(Free space)까지 물리 메모리에 그대로 할당**되므로 메모리 낭비가 심합니다. 또한 주소 공간이 물리적 메모리보다 큰 경우 실행하기 어렵습니다.

### 3. Segmentation (세그먼테이션)
Base-bound의 메모리 낭비 문제를 해결하기 위해 주소 공간을 **코드(Code), 힙(Heap), 스택(Stack)과 같은 논리적인 세그먼트 단위로 나누어** 메모리에 할당하는 기법입니다.
*   **주소 변환 공식:** 
    *   가상 주소는 최상위 몇 비트의 **세그먼트 비트(Segment bits)** 와 **오프셋(Offset)** 으로 나뉩니다.
    *   **`물리 주소 = Base[Segment] + Offset`**
    *   가상 주소의 오프셋이 해당 세그먼트의 Bound(크기)를 넘어서면 Segmentation Fault가 발생합니다.
*   **특징:**
    *   스택(Stack)은 거꾸로 자라므로(Grows backward), 오프셋을 계산할 때 음수 오프셋을 적용하는 추가적인 하드웨어 지원이 필요합니다.
    *   읽기/쓰기/실행 권한을 나타내는 보호 비트(Protection bits)를 두어 세그먼트 간 코드 공유 등이 가능합니다.
*   **장단점:** 메모리를 논리적 단위로 나누어 빈 공간의 낭비(내부 단편화)는 줄일 수 있으나, 프로세스가 필요로 하는 크기가 제각각이므로 메모리상에 다양한 크기의 빈 공간이 흩어지는 문제(외부 단편화)가 발생할 수 있습니다 (연속 메모리 할당의 특징).

### 4. Paging (페이징)
가상 메모리를 동일한 크기의 블록인 **페이지(Page)** 로 나누고, 물리 메모리를 이와 동일한 크기의 **페이지 프레임(Frame)** 으로 나누어 비연속적으로 메모리를 할당하는 기법입니다.
*   **주소 변환 공식:**
    *   가상 주소는 **VPN (가상 페이지 번호, Virtual Page Number)** 과 **Offset (페이지 내 오프셋)** 으로 나뉩니다.
    *   VPN을 인덱스로 삼아 **페이지 테이블(Page Table)** 을 참조하여 **PFN (페이지 프레임 번호, Page Frame Number)** 을 찾습니다.
    *   **`물리 주소 = <PFN, Offset>`** (PFN에 Offset을 이어 붙여 물리 주소 생성).
*   **가상 주소 구성 비트 계산식 (매우 중요):**
    *   예시: **32비트 가상 주소** 체계에서 **페이지 크기가 4KB**인 경우.
    *   **Offset 비트 수:** 페이지 크기가 4KB ($4096 = 2^{12}$ 바이트)이므로, 페이지 내 위치를 특정하기 위해 **12비트**가 필요합니다.
    *   **VPN 비트 수:** 총 32비트 중 Offset 12비트를 제외한 **20비트 ($32 - 12 = 20$)** 가 VPN을 나타냅니다.
    *   **페이지 테이블 크기:** VPN이 20비트이므로 총 $2^{20}$개(약 100만 개)의 페이지 테이블 항목(PTE)이 필요합니다. 각 PTE가 4바이트라면, **프로세스당 페이지 테이블 크기는 $2^{20} \times 4B = 4MB$** 가 됩니다.
*   **특징 및 장단점:**
    *   **장점:** 외부 단편화가 발생하지 않습니다. 빈 프레임 관리가 쉽고 메모리 할당 및 해제가 빠릅니다.
    *   **단점:** 
        1.  **메모리 참조 오버헤드:** 페이지 테이블 접근 1번, 실제 데이터 접근 1번으로 **메모리 접근 횟수가 2배**로 늘어나 속도가 느립니다.
        2.  **페이지 테이블 저장 공간 문제:** 프로세스마다 거대한 페이지 테이블을 유지해야 하므로 (예: 100개 프로세스면 400MB) **메모리 소모가 극심**합니다.
        3.  페이지 단위로 고정 할당하므로 내부 단편화(Internal fragmentation)가 발생합니다.

### 5. TLB (Translation Look-aside Buffer)
페이징의 가장 큰 단점인 **'메모리 접근 2배에 따른 속도 저하'를 해결**하기 위해 MMU 내부에 위치한 **하드웨어 캐시**입니다. 자주 쓰이는 VPN-PFN 변환 정보를 저장합니다.
*   **동작 방식:**
    *   **TLB Hit:** 가상 주소의 VPN이 TLB에 존재하면, 페이지 테이블 접근 없이 즉시 PFN을 얻어 물리 메모리에 접근합니다 (메모리 접근 1회).
    *   **TLB Miss:** TLB에 VPN이 없으면 물리 메모리의 페이지 테이블을 조회하고, 그 결과를 TLB에 업데이트한 뒤 명령어를 재실행합니다.
*   **특징:**
    *   **지역성(Locality):** 최근에 접근한 메모리에 다시 접근하는 시간적 지역성(Temporal Locality)과, 주변 메모리에 접근하는 공간적 지역성(Spatial Locality) 덕분에 TLB는 높은 Hit Rate를 달성할 수 있습니다.
    *   **문맥 교환(Context Switching) 문제:** 프로세스가 바뀌면 이전 프로세스의 TLB 정보는 무의미해집니다. 매번 TLB를 비우는(Flush) 비용을 줄이기 위해, TLB 항목에 **ASID(Address Space Identifier, 주소 공간 식별자)** 를 추가하여 여러 프로세스의 변환 정보를 구분합니다.

### 6. Multilevel Paging (다단계 페이징)
페이징의 또 다른 단점인 **'페이지 테이블이 차지하는 거대한 메모리 공간' 문제를 해결**하기 위한 기법입니다. 선형 페이지 테이블을 트리(Tree) 형태로 만듭니다.
*   **개념 및 특징:**
    *   페이지 테이블을 페이지 크기 단위로 자른 뒤, 모든 페이지 테이블 항목이 유효하지 않은(사용하지 않는) 빈 공간이라면 아예 메모리에 할당하지 않습니다.
    *   이 조각난 페이지 테이블들의 유효성을 추적하기 위해 최상위에 **페이지 디렉터리(Page Directory)** 라는 새로운 구조를 둡니다.
*   **주소 변환 방식 (2단계 페이징 예시):**
    *   가상 주소 32비트, 페이지 크기 4KB인 경우 (VPN 20비트, Offset 12비트).
    *   20비트의 VPN을 다시 두 부분으로 나눕니다. 각 페이지가 1024개($2^{10}$)의 PTE를 담을 수 있다면, 상위 **10비트(p1)는 페이지 디렉터리 인덱스**, 다음 **10비트(p2)는 내부 페이지 테이블 인덱스**로 사용합니다.
    *   **가상 주소 구성:** `p1(10비트) | p2(10비트) | Offset(12비트)`.
*   **장단점:**
    *   **장점:** 사용 중인 주소 공간에 비례해서만 페이지 테이블 공간을 할당하므로 **메모리 공간을 크게 절약**할 수 있습니다.
    *   **단점 (시간-공간 트레이드오프):** 페이지 테이블 접근을 위해 더 많은 간접 참조(Level of Indirection)가 발생하므로 TLB Miss 시 물리 메모리 접근 횟수가 더 늘어나 **시간 복잡도가 증가**합니다.

운영체제 중간고사 대비를 위해 앞서 설명해 드린 내용에 이어, 페이지 테이블의 크기 문제를 해결하기 위한 또 다른 대안인 **Hashed Page Table(해시 페이지 테이블)**과 **Inverted Page Table(역 페이지 테이블)**에 대해 상세히 정리해 드립니다.

### 7. Hashed Page Table (해시 페이지 테이블)
주로 32비트보다 큰(> 32 bits) 거대한 주소 공간을 가지는 시스템에서 페이지 테이블 크기를 관리하기 위해 흔히 사용되는 기법입니다.

*   **동작 방식:**
    *   가상 페이지 번호(VPN)를 해시(Hash) 함수에 통과시켜 해시 페이지 테이블의 인덱스로 사용합니다.
    *   해시 테이블의 각 항목은 해시 충돌(Hash collision)을 해결하기 위해, 동일한 위치로 해시되는 원소들을 연결 리스트(Chain) 형태로 관리합니다.
    *   주소 변환 시, 이 체인을 따라가며 가상 페이지 번호가 일치하는지 순차적으로 비교(Search)합니다.
    *   일치하는 항목을 찾으면 그곳에 저장된 물리 프레임 번호(PFN)를 추출하여 물리 주소를 완성합니다.

### 8. Inverted Page Table (역 페이지 테이블)
기존의 방식들이 '가상 페이지'를 기준으로 페이지 테이블 항목을 유지했던 것과 달리, 발상을 뒤집어 시스템의 **실제 메모리인 '물리적 페이지(프레임)'를 기준**으로 단 하나의 통합된 페이지 테이블만 유지하는 기법입니다.

*   **구조 및 특징:**
    *   시스템의 물리적 메모리 프레임마다 단 하나의 항목(Entry)만 가집니다.
    *   테이블의 각 항목은 해당 물리 페이지를 현재 **'어떤 프로세스(PID)'의 '어떤 가상 페이지'가 사용하고 있는지**에 대한 정보를 담고 있습니다.
    *   가상 주소에서 물리 주소로 변환하기 위해서는, 원하는 가상 페이지 번호가 물리 메모리의 몇 번째 프레임에 있는지 페이지 테이블 전체를 검색(Search)해야 합니다.
    *   이러한 탐색 오버헤드를 줄이기 위해(하나 또는 소수의 항목만 검색하도록 제한하기 위해) 역 페이지 테이블에서도 **해싱(Hashing) 기법을 함께 사용**합니다.
*   **장단점:**
    *   **장점 (공간 절약):** 각 프로세스마다 거대한 페이지 테이블을 개별적으로 만들 필요가 없으므로, 페이지 테이블을 저장하는 데 필요한 메모리 공간을 획기적으로 감소시킵니다.
    *   **단점 (시간 지연):** TLB Miss가 발생했을 때 주소 변환을 위해 테이블을 검색해야 하므로 주소 변환(탐색)에 걸리는 시간이 증가합니다.
